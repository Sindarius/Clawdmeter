1 · System Overview

How all the pieces fit together

Clawdmeter is a desk-side hardware widget that shows your Claude Code API usage in real time. The firmware runs on an ESP32-S3 AMOLED dev board; a host daemon on your PC polls the Anthropic API and pushes JSON to the device over BLE.

2 · AMOLED-1.8 Board Port

firmware commit f3ed242 — May 17 2026

The project originally targeted only the Waveshare ESP32-S3-Touch-AMOLED-2.16 (480×480, CO5300 panel). A second board was ported: the Waveshare ESP32-S3-Touch-AMOLED-1.8 (368×448 portrait). The hardware differences required significant code additions.

🖥 Display

2.16": CO5300 QSPI, 480×480, SCLK=GPIO 38, RST direct

1.8": SH8601 QSPI, 368×448 portrait, SCLK=GPIO 11, RST via XCA9554 IO expander

👆 Touch

2.16": CST9220 @ I2C 0x5A

1.8": FT3168 @ I2C 0x38. Inline FocalTech register reader — avoids GPLv3 Arduino_DriveBus library.

🔌 IO Expander (new)

XCA9554 / PCA9554 @ 0x20. Controls LCD_RST, TP_RST, audio amp enable. PWR button on EXIO4. Must init before display or touch.

🔘 Buttons

2.16": 3 buttons (GPIO 0, 18, AXP PKEY)

1.8": 2 buttons (GPIO 0 + XCA9554 EXIO4 PWR). No GPIO 18.

AMOLED-1.8 boot init order (critical):
─────────────────────────────────────────────────────────────────
 Wire.begin(SDA=15, SCL=14)
       │
       ▼
 io_expander_init()          ← releases LCD_RST, TP_RST from reset
       │                        (failure = display + touch silently dead)
       ▼
 gfx->begin()                ← SH8601 initialise via QSPI
       │
       ▼
 ft3168_init()               ← FocalTech touch init
       │
       ▼
 axp_init()   qmi_init()     ← PMU + IMU on same I2C bus
The UI automatically adapts its layout: at boot compute_layout() reads board_caps() (width, height, button count) and selects fonts, margins, and element positions accordingly. No compile-time layout conditionals needed.

3 · HAL & Device-Abstraction Refactor

firmware commit 2035121 — May 20 2026

The first board port was implemented using #ifdef BOARD_AMOLED_18 guards spread across six shared source files (~30 blocks total). This was replaced by a clean HAL layer so that adding future boards never requires editing shared code.

Before: #ifdef sprawl                   After: HAL + per-board folders
────────────────────────────────        ───────────────────────────────────────
main.cpp                                firmware/src/
  #ifdef BOARD_AMOLED_18                  hal/
    ft3168_read(&x, &y);                    display_hal.h  ← init/begin/tick/draw
  #else                                     touch_hal.h    ← init/read(x,y,pressed)
    cst9220_read(&x, &y);                   input_hal.h    ← is_held(PRIMARY|SECONDARY)
  #endif                                    power_hal.h    ← battery_pct/is_charging/…
                                            imu_hal.h      ← rotation_quadrant
ui.cpp                                      board_caps.h   ← W, H, button_count, has_*
  #ifdef BOARD_AMOLED_18                  boards/
    lv_font_t* big = &styrene_28;           waveshare_amoled_216/
  #else                                       display.cpp  ← implements display_hal.h
    lv_font_t* big = &tiempos_56;            touch.cpp    ← implements touch_hal.h
  #endif                                      …
                                            waveshare_amoled_18/
                                              display.cpp
                                              touch.cpp  io_expander.{h,cpp}
                                              …
                                            template/     ← copy to start a new port

PlatformIO's build_src_filter includes the shared code plus exactly one board folder per environment, so the compiler never sees competing implementations.

Key rule: no #ifdef BOARD_* in shared code

Optional features are guarded by BoardCaps (runtime struct) for behaviour differences, and BOARD_HAS_* macros (compile-time, defined only inside a board's own folder) for hardware that isn't wired up at all. Shared code never branches on board identity.

4 · UX Collapse — Single Screen + Hold-to-Pair

UX commit a6ed7d2 — Jun 2 2026

Originally the device had three navigable screens: Splash, Usage, and Bluetooth. The Bluetooth screen was removed; pairing is now a hold gesture on the PWR button from any screen.

Before

3 screens: Splash → Usage → Bluetooth

Short press cycles screens. Bluetooth screen shows pairing status explicitly.

After

1 primary screen: Usage (with BLE status line at top)

Short press: cycle brightness. Hold 3 s: toggle BLE pairing mode. Splash on boot (button press to dismiss).

Button event routing (after collapse):
───────────────────────────────────────────────────────────
 PWR short press (< 3 s, on release)  →  brightness cycle
 PWR long press  (≥ 3 s, edge)        →  toggle BLE advertising / pairing
 BOOT short press                     →  Space key (HID keyboard)
 BOOT hold                            →  voice-mode trigger

 Splash screen: any button press      →  advance to Usage screen
───────────────────────────────────────────────────────────

The BLE status is now shown as a compact one-line indicator at the top of the Usage screen (connection state + device name + MAC address) rather than a dedicated screen, freeing the full display area for usage data.

5 · User-Controlled Brightness (NVS-Persisted)

firmware commit ddb812a — May 24 2026

Before: display brightness was a compile-time constant (200/255). After: the user can cycle through four levels via a short press on the PWR button, and the chosen level persists across power cycles via ESP32 NVS flash storage.

brightness.{h,cpp}   — new module
─────────────────────────────────────────────────────────
 Ramp: { 64, 128, 200, 255 }   (default = 200, same as old constant)

 brightness_get()      → current level (read by idle + display modules)
 brightness_next()     → advance ramp, save to NVS, apply via idle
 brightness_set(n)     → direct set (used on boot restore)

idle.cpp changes:
  Old: fade DISPLAY_DEFAULT_BRIGHTNESS (constant) → 0 on idle
  New: fade brightness_get() → 0 on idle; restore brightness_get() on wake
       (user choice survives idle fade/wake cycles)

display.cpp changes (AMOLED-216):
  Old: post-rotation ramp targeted hard-coded 200
  New: ramp targets brightness_get() so rotation doesn't reset user choice

6 · Power HAL Long-Press Edges

firmware commit 98a513a — May 24 2026

The hold-to-pair gesture (§4) requires the HAL to expose a "long-press" edge event from the PWR button. Each board implements this differently.

AMOLED-2.16 (AXP2101 PKEY)

Enables AXP2101 PKEY LONG + POSITIVE IRQs. Hardware shutdown threshold bumped to 8 s so a 3–6 s hold gesture doesn't accidentally power off the device.

AMOLED-1.8 (XCA9554 EXIO4 poll)

No hardware IRQ available. Long/release edges are synthesised in software from the polled GPIO level. Short-press fires on release-if-short to avoid false triggers during a hold.

Two new functions added to power_hal.h:

+ power_hal_pwr_long_pressed() → bool // rising edge: held ≥ 3 s
+ power_hal_pwr_released() → bool // falling edge after short press

7 · Rotation Flash Brightness Fix

bug fix commit 65e51a0 — May 30 2026

The CO5300 AMOLED can't rotate natively; the firmware does CPU pixel remapping in strips, then briefly blanks and re-brightens the panel to avoid tearing artefacts. This ramp was hardcoded to 200/255, so rotating the device reset whatever brightness the user had set.

boards/waveshare_amoled_216/display.cpp - target_brightness = 200; // hard-coded old default
+ target_brightness = brightness_get(); // user's chosen level

The 13 pixel-art Claude creature animations sourced from claudepix.vercel.app use a body colour of #CD7F6A. This was remapped in the conversion pipeline to the official Anthropic brand terracotta #D97757.

9 · BLE Device Rename: "Claude Controller" → "Clawdmeter"

firmware daemon commit 13cc423 — May 26 2026

For brand consistency the BLE advertised name was changed from "Claude Controller" to "Clawdmeter". Three functional locations must agree for discovery-by-name to work — the firmware, the Linux/macOS daemon, and the Windows daemon.

firmware/src/ble.cpp - #define DEVICE_NAME "Claude Controller"
+ #define DEVICE_NAME "Clawdmeter"
daemon/claude_usage_daemon.py (macOS/Linux) - DEVICE_NAME = "Claude Controller"
+ DEVICE_NAME = "Clawdmeter"
daemon/claude-usage-daemon.sh (Linux Bash) - DEVICE_NAME="Claude Controller"
+ DEVICE_NAME="Clawdmeter"
Pairing cache note: Hosts cache the GATT name. An already-paired device keeps showing the old name until you remove it from the Bluetooth manager (bluetoothctl remove <mac> on Linux, or "Forget" in System Settings on macOS/Windows) and re-pair.

10 · macOS Daemon — retrieveConnected Fix

daemon bug fix commit 18282e0 — May 30 2026

On macOS the ESP32 firmware advertises a BLE HID keyboard profile. macOS auto-connects to it and holds the HID link. CoreBluetooth excludes already-connected peripherals from scan results, so the daemon's scan loop never found the device — it was stuck forever with "Device not found" even though the device was visible in System Settings.

Old flow (broken on macOS):
────────────────────────────────────────────────────────────
  BleakScanner.find_device_by_name("Clawdmeter")
        │
        ▼
  CoreBluetooth scan result list
        │   ← ESP32 already held by HID, excluded from results
        ▼
  None  →  "Device not found"  →  retry forever

New flow (macOS):
────────────────────────────────────────────────────────────
  CoreBluetooth.retrieveConnectedPeripheralsWithServices_([custom UUID])
        │   ← finds already-connected peripherals directly
        ▼
  match custom service UUID  →  connect directly (no scan)
        │
        │  fallback if not found via custom UUID:
        ▼
  match HID service 0x1812 + exact DEVICE_NAME  →  connect
        │
        ▼
  daemon GATT write channel runs alongside macOS HID link (two slots)
────────────────────────────────────────────────────────────
Linux/BlueZ path: unchanged (scan + address cache)

11 · Windows Daemon (Brand New)

daemon commit eb70456 — Jun 2 2026

Before this change there was no native Windows support — users on Windows had to use WSL. A complete Windows port was added: a Python daemon using bleak (WinRT BLE backend) with a system-tray UI via pystray.

claude_usage_daemon_windows.py

Reads OAuth token from %APPDATA%\Claude\claude_desktop_config.json (no keychain). Polls Anthropic API. BLE scan/connect/write with full auto-reconnect logic.

tray_windows.py

pystray system-tray icon. Runs daemon loop in background thread. Status icons reflect connection state. Quit handler shuts daemon cleanly.

autostart_windows.py

Writes HKCU\Software\Microsoft\Windows\CurrentVersion\Run entry via winreg so the tray starts at logon under pythonw.exe (no console window).

Resilience constants

CONNECT_RETRIES=3, ZOMBIE_BREAK_LIMIT=1 (drops half-open BLE link after first consecutive write failure), RECONNECT_BACKOFF_CAP=8 s.

Windows daemon reconnect state machine:
──────────────────────────────────────────────────────────────
         ┌─────────────┐
         │  SCANNING    │◄──────────────────────────────┐
         │ (8 s window) │                               │
         └──────┬───────┘                               │
                │ found                                 │ scan miss
                ▼                                       │ or zombie break
         ┌─────────────┐   write failure (N=1)          │
         │  CONNECTED   ├───────────────────────────────┘
         │  (poll 60 s) │
         └──────┬───────┘
                │ disconnect detect
                ▼
         ┌─────────────┐
         │  BACKOFF     │  exp. backoff, cap = 8 s
         │  (fast-retry)│──────────────────────────► SCANNING
         └─────────────┘
──────────────────────────────────────────────────────────────
Rotating log: %LOCALAPPDATA%\Clawdmeter\daemon.log (512 KB × 3)

12 · AMOLED-1.8 Hardware Revision Auto-Detection

firmware commit 4e54363 — Jun 3 2026

Waveshare silently revised the AMOLED-1.8 hardware. The original board ships with SH8601 display + FT3168 touch (I2C 0x38). A later revision ships with CO5300 display + CST816 touch (I2C 0x15). Both boards are sold under the same product name with no external way to tell them apart.

The original firmware assumed the first revision, leaving the CO5300 panel blank and spamming I2C errors.

board_init.cpp — probe at boot:
──────────────────────────────────────────────────────────────────
  Wire.beginTransmission(0x15)   ← CST816 (Rev 2 touch)
  Wire.endTransmission()
        │
        ├── ACK (0x00)  →  board_rev = REV_CO5300_CST816
        │                  display: Arduino_CO5300 + col_offset=16
        │                  touch: FocalTech reader @ 0x15
        │
        └── NACK        →  board_rev = REV_SH8601_FT3168
                           display: Arduino_SH8601
                           touch: FocalTech reader @ 0x38

One binary, two hardware revisions.
──────────────────────────────────────────────────────────────────
CO5300 note: its 368-wide active area sits at GRAM column 16,
so CO5300_COL_OFFSET=16 is added to all draw calls to center it.

board_rev.h (new)

Defines BoardRev enum: REV_SH8601_FT3168 and REV_CO5300_CST816. Exposes board_rev() accessor.

display.cpp + touch.cpp

Both switch on board_rev() at runtime. A single Arduino_OLED* base pointer holds either display driver. Touch uses the same FocalTech register layout (0x02–0x06) for both chips — only the I2C address differs.

13 · Today's Fix — Windows DEVICE_NAME Mismatch

bug fix daemon diagnosed & fixed Jun 4 2026

The Windows daemon was added (§11) before the BLE rename (§9) was merged into the Windows branch. When the PR landed, the Windows daemon's DEVICE_NAME was never updated to match the firmware.

Symptom: tray app is running, BLE device is powered on and connected via USB — but the device never receives any data. No log file was created in %LOCALAPPDATA%\Clawdmeter\ either, confirming the scan loop never found the peripheral and the daemon thread was silently spinning.

Discovery-by-name requires three values to match:
─────────────────────────────────────────────────────────────────────
  firmware/src/ble.cpp
    #define DEVICE_NAME "Clawdmeter"     ← what the ESP32 advertises

  daemon/claude_usage_daemon_windows.py
    DEVICE_NAME = "Claude Controller"    ← ✗ MISMATCH — never found

  daemon/claude_usage_daemon.py (macOS/Linux)
    DEVICE_NAME = "Clawdmeter"           ← ✓ already correct (from §9)
─────────────────────────────────────────────────────────────────────
daemon/claude_usage_daemon_windows.py (line 26) - DEVICE_NAME = "Claude Controller"
+ DEVICE_NAME = "Clawdmeter"

After restarting the tray app the Windows daemon scans for the correct name, finds the device, connects, and begins pushing usage data.
